# DECISIONS.md

Architecture decisions and the reasoning behind them, in case future sessions are tempted to re-decide. Moved here from STATE.md (which stays a short "where am I" snapshot). Organized by the stage/feature that produced them; the decisions themselves are durable.

For the hard rules distilled from these (standing constraints, things NOT to do), see CLAUDE.md. For current status, see STATE.md.

## Architecture decisions made (and the reasoning, in case I forget)

### From earlier stages
- **Vertical-stack-of-typed-blocks editor** (Notion-style), not grid canvas.
- **Tiptap on ProseMirror, with React bindings**, KaTeX for math. Vue 3 was the close runner-up.
- **Tailwind v4 via the `@tailwindcss/vite` plugin**. Single `@import "tailwindcss"` in `src/index.css`. No PostCSS, no `tailwind.config.js`. Theme customization in CSS via `@theme`.
- **React Router v7** for SPA routing.
- **`/playground` is dev-only** — route registered conditionally on `import.meta.env.DEV`. Production bundle physically excludes it.
- **Editor lives as shared module at `src/editor/`**, used by `/playground` and `/activity/:id`.
- **Static HTML on publish** (option C). Host: Cloudflare R2 (Stage 13.6+); initially attempted Supabase Storage but Supabase rewrites HTML responses to text/plain.
- **Two URLs per published activity:** live alias (`/index.html`, 5min cache) and immutable versioned permalink (`/v{N}/index.html`, 1yr cache).
- **Drafts in a separate column** (`activities.draft_content`); versions append-only.
- **Permission helpers as SQL functions** so Phase 3+ collaboration changes one function, not eight policies.
- **Submission identity always present** — Pattern B: name field on the page, gate submit until non-empty. CHECK constraint enforces at storage layer.
- **Responses jsonb keyed by stable `blank.id`**, not array index.
- **`schemaVersion: 1`** on `ActivityDocument` stays at 1 — Stage 9a additions are all optional-with-defaults, so existing stored documents parse cleanly. `SubmissionResponses.schemaVersion` is at 2 with migrate-on-read for v1.
- **`attempt_number` lives on the submissions table as a column**, derived server-side via the `ingest_submission` RPC (`max + 1` over the student's identity scope, with `unique_violation` retry against partial unique indexes added in 0005). The Edge Function returns the canonical value in its HTTP response so the runtime can reconcile its optimistic guess. Client value is advisory only.
- **Runtime answer scoring is strategy-dispatched** — `evaluateAnswer(input, typed)` reads `data-blank-strategy`, defaulting to `'list'`.
- **Math nodes are atoms** with `attrs.latex` as the source of truth. Serialize as `<span data-math-inline data-latex="...">` / `<div data-math-block data-latex="...">`.
- **NodeView lifecycle pattern (the 5 commitments):** `useLayoutEffect` (synchronous before paint), stable ref as render target, tight dependency arrays (`[latex]`), `throwOnError: false`, `NodeViewWrapper as="span"` for inline / `as="div"` for block.
- **React state vs ProseMirror state in NodeViews:** UI concerns → React `useState`. Document concerns → ProseMirror (`updateAttributes`, `editor.isActive`). Conflating them was the cause of the "input deselects after one keystroke" bug.
- **Math NodeView edit-mode focus.** Entering edit mode must explicitly `.focus()` the `<math-field>`, deferred one animation frame (MathLive custom elements don't autofocus, and a focus issued the same tick the element mounts can be dropped). Exiting edit mode is wired to the field's `onBlur`.
- **MathLive in published HTML is a Phase 2.5 decision**, not committed. Editor-only commitment now.
- **Toolbar UX: static (always-visible) over BubbleMenu/FloatingMenu** — discoverability beats elegance for non-technical teachers.
- **Slash menu architecture:** `@tiptap/suggestion` + `tippy.js` + Tiptap's `ReactRenderer` (the `useImperativeHandle` pattern). Menu items live in `slashMenuItems.ts` as a flat array.
- **Drag handle: Notion-style, mouse-only** via `@tiptap/extension-drag-handle-react`. Hover-revealed in left gutter.
- **Keyboard reorder via `Mod+Shift+ArrowUp/Down`** — custom `BlockReorderShortcuts` extension. Operates on whichever top-level block contains the cursor.
- **`Editor` exposes `onUpdate(json)` callback prop** as the single hook for "react to doc changes."
- **JSON inspector lives in `Playground.tsx`, not `Editor.tsx`.** Standard "lift state up."
- **Serialize layer pattern:** `serialize.ts` in `@activity/app/src/lib/`, exports `tiptapToActivity` / `activityToTiptap`. The renderer never touches this file.
- **Three submission modes:** `single` (no checkpoints), `locked` (checkpoints freeze on check), `free` (checkpoints allow revision). Activity-level field on `ActivityMeta`, default `free`. Per-section checkpoints — tied to section boundaries.
- **Two revision modes:** `free` (revise + resubmit) and `locked` (no revision/resubmit). Activity-level, default `free`. Separate from submissionMode — `single` mode ignores `revisionMode`.
- **Attempt tracking:** each resubmit is a new `submissions` row with incremented server-derived `attempt_number`. Teacher dashboard has toggle: all attempts vs best score + count.

### Runtime architecture (Stages 11–12)
- **Runtime: real modules in source, inlined in output.** Source at `packages/renderer/src/runtime/`; built by `scripts/bundle-renderer.mjs` as a first esbuild step (minified IIFE, `chrome90` target) into a generated TypeScript string module (`runtime/generated/runtime-bundle.ts`), which `document.ts` imports and inlines into every published page via a `<script>` tag. Inlining is the right call for Phase 1: one fewer request on a slow Chromebook, runtime stays co-versioned with the HTML it ships in. Separate-file CDN model is a Phase 3+ migration target.
- **Two tsconfigs in the renderer package.** The renderer is pure no-DOM TypeScript; the runtime is DOM TypeScript. `packages/renderer/tsconfig.json` excludes `src/runtime/**`; `packages/renderer/src/runtime/tsconfig.json` adds the DOM lib with `noEmit` (esbuild does the build). The package's `build` and `typecheck` scripts run both `tsc` invocations in sequence so runtime cannot silently rot.
- **Data-attribute contract is a public API.** Renderer emits; runtime reads. Frozen for already-published activities; additive changes only. Every attribute read uses `?? default` fallback. Init and event handlers wrapped in try/catch — graceful degradation to basic submit if init fails.
- **Renderer data-attribute contract.** Blank tokens are `<input class="blank">` inside `<span class="blank-wrapper">` with sibling `.js-blank-feedback` (void-element-can't-have-children fix); always-available hint `?` button (not post-incorrect reveal); mistake feedback re-evaluated each check; `data-submission-mode` deliberately NOT on the container in single mode (checkpoint markup omitted entirely); split-by-purpose contract (CSS hooks → `data-*` on `.activity-container`, JS-only config → six-field `#activity-config` blob); per-block (not per-blank) confidence fieldset.
- **`init.ts` is the only DOM walker.** Returns `{config, refs, state} | null`; parseConfig returns null on missing/malformed config and the caller falls back to no-op runtime. `buildRefs` walks sections → fill-in-blank blocks → blanks once and produces typed `BlankRef` / `FillInBlankRef` / `SectionRef` maps keyed by uuid. Downstream code consumes refs; nothing else calls `querySelector` against the body. Malformed per-element data attributes are warn-and-skip, not throw.

### Runtime feedback machinery + persistence (Stage 13)

- **State→DOM separation: `render(state, refs)` in `render.ts` is the only function in the runtime that mutates the DOM after init.** Every event handler mutates `RuntimeState` then calls a unified `onUpdate` callback that runs render + persist. Concentrating the DOM-mutation exception in one file makes the rule grep-able and enforceable. One exception: `applyStoredState` in `storage.ts` sets `input.value` during bootstrap restoration, BEFORE the initial render runs and before handlers attach.

- **Change-guard pattern in render.** Every DOM write checks the current DOM state and only writes when the target differs. For class toggles, `classList.toggle(name, condition)` is idempotent. For attribute / `hidden` / `checked` / `textContent` writes, explicit current-vs-target checks. Renders are idempotent — calling twice with the same state produces no observable diff.

- **State shape (per-entity, all populated by `createInitialState`):**
  - `BlankState`: `result` (true/false/null), `matchedMistake` (string or null). (No `hintRevealed` — hint visibility is now global modal state, see below.)
  - `BlockState`: `solutionRevealed` (boolean), `confidence` (`'unsure' | 'think_so' | 'certain' | null`).
  - `SectionState`: `checked`, `locked`, `score`, `total`, `checkedAt`. `total` is section total (denominator for score display), NOT attempted count.
  - Top-level `RuntimeState.hintModalBlankId` (`string | null`): which blank's hint the single global modal is showing, or null when closed. Not persisted.

- **`scoreBlankAndUpdateState` renamed from `checkBlank`.** Pre-Stage-13, `checkBlank` mutated DOM via `applyBlankFeedback`. Post-Stage-13, the function mutates state only — render handles DOM. The rename makes the side effect explicit. Takes `(state, id, ref)`; id passed explicitly so the function doesn't read it from DOM (`ref.input.dataset.blankId` would violate the init-walks-DOM rule). Caller has id from Map iteration.

- **`clearBlankState` returns boolean change indicator.** Used by the `input` event handler to skip `onUpdate` (render + persist) on keystrokes that don't actually change state — avoids cascading renders during fresh typing. Returns true only when result or matchedMistake was non-null and got cleared.

- **Mistake feedback matching rules.** String match against `BlankRef.mistakeFeedback` entries' `match` field. Case-insensitive: trim AND lowercase both sides before compare (a student shouldn't lose targeted help over capitalization). First match wins. Computed at scoring time (in `scoreBlankAndUpdateState`) and stored in `BlankState.matchedMistake`; render reads it.

- **Feedback slot policy.** `.js-blank-feedback` carries mistake-specific text only — populated when a wrong answer matches an authored mistake entry, hidden otherwise. Visual correct/incorrect signal lives on the input border class (`.correct` / `.incorrect`). Screen reader signal is `aria-invalid="true"|"false"` on the input itself (removed entirely when result is null, so the attribute doesn't claim "this is valid" before the student has tried). Avoids "Correct ✓" clutter on a 30-blank worksheet.

- **Hint affordance (global modal).** A single hint modal element lives at the end of `<body>` (emitted by `document.ts`); `refs.hintModal` is its `HintModalRef`. Clicking a blank's `?` button sets `RuntimeState.hintModalBlankId` to that blank's id; `render` copies the blank's `data-hint` into the shared modal body, unhides it, and flips that blank's button `aria-expanded`. Setting `hintModalBlankId` back to null closes it. Replaced the old per-blank inline-reveal (`hintRevealed` + `.js-blank-hint-text` span) — one modal gives more room and future rich formatting. Modal open state is NOT persisted across reload (transient UI, unlike scored results).

- **Edit-to-clear behavior.** `wireBlanks` attaches both `blur` (score + render) and `input` (clear stale state + conditional render) handlers. When the student types into a blank that already has a scored result, the input handler clears it so a stale green "correct" border doesn't linger on an edited but un-re-blurred answer.

### Fill-in-blank authoring (Stage 13.5)

- **Fill-in-blank block is a `fillInBlank` Tiptap node** with content spec `(text | mathInline | blank)*`. Inserted via slash menu, the "Problem" toolbar button, or markdown sentinel `{{answer|alt|alt}}` parsing.
- **Blanks are inline atom nodes living ONLY inside fillInBlank blocks** (the schema's `FillInBlankInline` union: text | mathInline | blank). The block's content spec enforces placement at the ProseMirror level.
- **Three insertion paths**: slash menu, sentinel `{{...}}` typed inline (pipe-delimited acceptable answers — matches runtime's `data-blank-answers` format), or "Blank" toolbar button (enabled only when cursor inside a fillInBlank).
- **Chip rendering**: a small inline pill with the canonical answer + underline beneath. Width derived from answer length: `Math.max(answer.length + 1, 4)` ch. Formula must stay in sync between `BlankView.tsx` (editor) and `inline.ts` (renderer).
- **Per-blank editing via a single popover at editor root**, not per-chip. `BlankPopoverHost` watches editor selection (via `instanceof NodeSelection` narrowing); renders `BlankEditPopover` when a `blank` node is the active selection. One floating-ui instance, one portal mount/unmount lifecycle tied to selection. Per-chip popover was tried in Drop 1 and broke widespread editor behavior (slash menu inline-render, fill-in-blanks uneditable, drag handles broken) — diagnosed as React reconciliation issues with N permanently-mounted popovers.
- **Popover state lifecycle**: local React state for editing, refs mirror state for synchronous flush reads, initial-value refs hold the baseline for change diffs. Per-field save-on-blur is the normal commit path; `flushAll()` runs on every close path (Escape, outside click, Enter) so typed-but-not-blurred values survive close.
- **`updateBlankAttrs(pos, attrs, options?)`** — chain command for updating blank attrs by position. `options.preserveSelection` (default `true`) re-applies NodeSelection at the chip's position after setNodeMarkup so the popover stays open across edits. Close-time flushes pass `false` so the subsequent setTextSelection in onClose can move selection cleanly off the chip in one click. Without the flag, the re-asserted selection fights with onClose's selection move and the popover bounces back open, requiring a second click.
- **`onChange` thread carries options through**: BlankEditPopover → BlankPopoverHost.handleChange → editor.commands.updateBlankAttrs. Lets the popover request `preserveSelection: false` when flushing on close.
- **Popover positioning**: floating-ui with `offset`, `flip`, `shift`, and `size` middleware. `size` dynamically computes `max-height` based on available viewport space at the chosen placement, floor `MIN_POPOVER_HEIGHT = 200`. `flip` lets the popover anchor above the chip when there's more space there (tall popovers near page bottom flip up). CSS no longer sets `max-height` — it's controlled by JS via inline style.
- **Focus trap via `focus-trap-react`**. Tab cycles within the popover; Escape and outside-click still fire our handlers (`escapeDeactivates: false`, `allowOutsideClick: true`). Initial focus deferred to our rAF-based answer-field focus rather than FocusTrap's synchronous default. Focus returns to the chip on close (`returnFocusOnDeactivate: true`).
- **Document-level outside-click handler** in BlankEditPopover closes the popover when mousedown lands outside both popover and chip. Covers page-chrome clicks where ProseMirror's selection-change wouldn't fire.
- **Toolbar buttons for insertion**: Three new buttons across two groups.
  - "Problem" (always enabled, inserts empty `fill_in_blank` block)
  - "Blank" (enabled only when `editor.isActive('fillInBlank')`, inserts blank with `?` placeholder, then `setNodeSelection` via rAF so the popover auto-opens for immediate editing)
  - "Section" (in a separate Structure group at toolbar end, inserts section break via `insertSectionBreak()` — SectionBreakView's existing inline UI handles title + checkpoint editing)
- **Serialize layer extended for Stage 13.5**: `tiptapBlankToActivity` and `activityBlankToTiptap` round-trip all four blank fields (answer, acceptableAnswers, hint, mistakeFeedback). Optional fields (hint, mistakeFeedback) only serialized when non-empty for round-trip exactness.
- **`FillInBlank` carries `definingForContent: true`.** Prevents ProseMirror's content-fit algorithm from auto-lifting an inline atom out of a parent block it just emptied — root cause of the empty-block disappearing on the sentinel input rule. `defining: true` would also fix that symptom but breaks drag-reorder asymmetrically (it doubles as `definingAsContext`, which preserves drag-source context too aggressively and makes later blocks un-droppable above earlier ones). `definingForContent` is the targeted granular flag that isolates the destination-side preservation we want without the source-side regression. The Blank input rule handler passes `range` directly to `insertContentAt` (cleaner than the prior recomputed `from`/`to`, though that recomputation wasn't itself the bug).
- **Clicking another blank chip while a popover is open: SWITCH-vs-CLOSE is a known, deferred design decision (2026-06-16).** The user-friendly target is a one-click switch — clicking chip B while editing chip A should retarget the popover to B (flushing A first). Today it instead *closes* A's popover (committing A); a second click opens B. **There is no data loss** — A's edit always commits via the outside-click `flushAll` plus the field's blur (verified by reproduction: type in A → click B, including rapid/synchronous variants; A's value persists). The wart is purely the extra click. **Why it's still deferred:** a one-click-switch was attempted twice and reverted — (1) early-returning from the outside-click handler on a `.blank-chip` target left focus trapped in the popover so the selection never moved to B (stayed on A); (2) adding an explicit host `setNodeSelection(B)` after `flushAll` also failed, because `flushAll`'s `updateBlankAttrs` transaction (`preserveSelection: false`) disrupts the in-flight selection change and the `FocusTrap` holds focus. This is the same selection/FocusTrap entanglement that broke widespread editor behavior in Drop 1, so it needs a dedicated design pass (e.g. a deliberate close-then-reopen that frees the trap, or restructuring the flush so it doesn't fight the retarget) rather than a quick patch. Low priority precisely because there's no data loss.

### Publish flow infrastructure (Stage 13.6)

- **PublishControl placement: page header, NOT editor toolbar.** Publish is an activity-level action; the toolbar is for editor-formatting controls. Sits right of `SaveIndicator` in `ActivityEditor.tsx`. Mirrors common patterns (Google Docs, Notion both put "Share" top-right).
- **No confirmation modal for first publish or republish.** Each publish creates a new immutable version row; nothing destructive happens. Friction cost of a modal isn't justified yet. Easy to add later if version sprawl becomes a real problem.
- **Disabled while `saveStatus === 'saving'`.** Prevents publishing a stale draft mid-autosave. The 1s useAutosave debounce window can still race a fast click; relying on the "Saved" indicator for now. Proper `flush()` on useAutosave is Stage 14 polish.
- **"Publish" → "Publishing..." → "Republish" label progression** in-session. The "Republish" wording surfaces only after the first successful publish in the current session — reduces "wait, did I already do that?" moments without requiring a persistent state read on mount.
- **Success state shows the live URL inline.** Green pill displays `Published v{N}` + Copy URL button + Open-in-new-tab link. The Open link is the critical affordance — it's how the teacher previews as a student. Public URL (live alias) shown; versioned URL recorded internally but not surfaced for routine sharing.
- **Editor load priority: draft → current version → fresh empty.** The publish RPC clears `draft_content` atomically on success. Without the version-content fallback, reloading the editor after publish shows an empty doc (real bug, first reported during Stage 13.6 sketching, fixed same session). Both queries are issued from the same `useEffect` with cancellation guards. `Activities.tsx` always inserts a draft on create, so the fresh-empty branch is defensive (theoretically unreachable via the normal flow).

### Hosting platform (Stage 13.6)

- **Published HTML hosts on Cloudflare R2, NOT Supabase Storage.** Supabase free-tier prohibits HTML content from being served on `*.supabase.co` domains:
  - **Storage**: public buckets serve HTML as `Content-Type: text/plain` with a sandbox CSP, as an anti-abuse measure to prevent the platform from being used for arbitrary web hosting. Documented at https://supabase.com/docs/guides/storage/quickstart.
  - **Edge Functions**: same rewrite is applied at the gateway level — `GET` requests returning `text/html` are downgraded to `text/plain` with the same sandbox CSP, regardless of what the function's `Response` constructor sets. Documented at https://supabase.com/docs/guides/functions/http-methods and https://supabase.com/docs/guides/functions/limits.
  - Only exception is Pro-plan with custom domain ($25/mo); ROADMAP's sustainability model defers paid tiers to Phase 4.
- **Discovered during Stage 13.6 e2e testing.** First attempt was direct Storage public URL → browser saw text/plain, won't render. Second attempt was a `serve-activity` Edge Function proxy that reads from Storage and re-emits with the right Content-Type → same gateway rewrite hit at the function-response level. Both architectures dead-ended for the same root cause.
- **Cloudflare R2 is the chosen replacement.** Reasoning:
  - **Zero egress cost** (the unique R2 feature; every other major object store charges per-GB downloaded). For a K-12 platform with student-driven download volume, this scales sustainably to Phase 5 marketplace.
  - **Generous free tier:** 10 GB storage, 1M Class A operations (writes), 10M Class B operations (reads) per month. Fits Phase 1-3 by a wide margin; even Phase 4 district scale comfortably.
  - **S3-compatible API.** Standard AWS SDK works in Deno via esm.sh. No proprietary library lock-in; migration to another S3-compatible bucket (Backblaze B2, Wasabi, AWS S3) later is a config-and-DNS change, not a code rewrite.
  - **Free custom domain support.** Phase 4 district procurement conversations get easier when activity URLs live at `activities.yourbrand.com` rather than `pub-7a3f.r2.dev`.
  - **ROADMAP-aligned.** Already named as the destination in Phase 2/3; pulling it forward to Phase 1 is acceleration, not deviation.
- **Hybrid architecture: keep Supabase for everything else.** Database, auth, Edge Functions (publish-activity, ingest-submission) all stay on Supabase. R2 hosts ONLY the published HTML output. This is the minimum-viable migration; full Cloudflare migration (Workers + R2 + Pages) is a future option but not warranted now.
- **`serve-activity` Edge Function: deleted.** Was created in this session as the proxy attempt; superseded by R2 (which serves HTML directly with the right Content-Type). Local directory removed; Supabase deployment removed via `supabase functions delete serve-activity`.

### Editor UI for feature fields (Stage 15)

- **Activity metadata was already persisted before Stage 15.** `ActivityEditor` held the full `ActivityMeta` in state and round-tripped it through `draft_content`; only `title` had a control. The other fields loaded and saved back unchanged. So adding pickers was pure wiring to `setMeta` — `changeKey` already fingerprinted meta, so autosave needed no change.
- **Activity settings live in a collapsible disclosure under the title, NOT a modal/gear-drawer.** Discoverability-beats-elegance (same reasoning as the static toolbar over BubbleMenu). Always-visible label, expand for the three pickers.
- **`revisionMode` control is disabled (not hidden) in single mode**, with "Not used in single-submit mode." helper text. Hiding it would make the submissionMode↔revisionMode coupling invisible; disabling makes it legible.
- **`gradingMode` gets no picker in Phase 1.** It's inert (manual/mixed treated as auto until Phase 2.6 per-block grading lands). Surfacing a control would imply functionality that doesn't exist. Add it when manual-graded block types arrive.
- **`activityType` picker is included even though renderer presentation-branching is Phase 2.** It has a real near-term consumer: Stage 16's dashboard filters on it.
- **Locked-mode checkpoint validation WARNS, does not block save.** Blocking save mid-edit is hostile. An amber inline banner (continuous, not just at publish) fires whenever locked mode coexists with any non-checkpoint section. The implicit leading section (content before the first `sectionBreak`) counts as non-checkpoint. Resolves open-question #258.
- **FillInBlank block-field UI is an inline NodeView footer disclosure, NOT a popover.** Mirrors SectionBreakView's inline-controls pattern and sidesteps the per-chip-popover hazards documented in Stage 13.5. The blank-token popover stays for per-blank fields; block-level fields (solution, confidence) get the footer. The footer stays hidden for a plain/unselected/unconfigured problem to keep long worksheets uncluttered (shown when `selected || settingsOpen || isConfigured`). It's `contentEditable={false}` so ProseMirror doesn't treat it as block content; the solution textarea `stopPropagation`s keydown so editor shortcuts don't fire while typing.
- **`solution` was a plain multiline string in Phase 1; Phase 2 upgraded it to inline rich text + math** (now `InlineNode[]`, authored via `InlineRichTextEditor` in the footer — see Current focus).
- **`skills` editing UI deferred to Phase 2** (the schema scheduled it there). The attr round-trips through the node and serialize so data survives, but no control surfaces it. Don't add a skills tag-input without revisiting Phase 2 scope.

### Print feature (Drops A–D)

- **One renderer, not two.** Print is the same `renderBody` path as the screen; print config produces classes / CSS vars / data-attrs consumed under `@media print`. `renderActivity` (published page) and `renderActivityForPrint` (print document, no interactive chrome) share `renderBody`.
- **`@page` can't read CSS custom properties.** So paperSize + margin go out as a literal per-document `<style>@page{size;margin}</style>` (`printPageStyle`); columns/workSpace/fontSize/problemSpacing ride as `--print-*` vars on `.activity-container` where `@media print` selectors read them.
- **Additive optional-with-defaults schema → no `schemaVersion` bump, no migration.** `meta.print` round-trips through `draft_content` for free because `tiptapToActivity` embeds `meta` whole; only per-block `workSpace` needed a Tiptap attr + serialize handling.
- **Print confidence is a paper twin.** The interactive `.js-confidence-rating` fieldset is print-hidden; a separate `.print-confidence` hand-markable checkbox row (screen-hidden, print-shown) renders only for confidence-rating problems. Don't try to print the interactive control.
- **The print route is client-side (Route 2-friendly).** The app imports `@activity/renderer` directly via Vite, so `/activity/:id/print` and the foldable need no Edge Function and no redeploy. `showAnswers` is print-route-only and never reaches `publish-activity`, so its bundle is server-irrelevant.
- **Journal foldable = client-side, DOM-measured (Route 2); never on the published static page.** It needs a live DOM to measure block heights, so it lives only in the app print route. Engine split mirrors the physical process: **pagination** (chunk blocks into fixed-height panels, never splitting a problem) + **composition** (arrange panels onto duplex sheets). Both are pure + unit-tested (`packages/app/src/lib/foldable/{paginate,compose}.ts`); `measure.ts` (offscreen iframe) and `render.ts` (assemble) are the DOM/output glue; `index.ts` orchestrates.
- **Measure === print is the load-bearing invariant.** The offscreen measuring iframe and the printed document share ONE stylesheet (`foldable/styles.ts`), included after the renderer's `blockStyles`, which **promotes the renderer's `@media print` rules to unconditional** (paper-twin confidence shown, interactive controls hidden, blanks→underlines) and zeroes block margins in favor of one controlled flow gap — so a measured border-box height + one gap fully describes the stack, and what's measured is what prints. Geometry is authored in inches; measurement comes back in px at the CSS-fixed 96px/in, and the root font-size is pinned to 16px so `rem` resolves identically in both contexts.
- **Booklet geometry (confirmed from a real artifact).** Single vertical fold, landscape sheet, double-sided → 4 half-sheet panels (2 per duplex face). Reading order `[content #1, #2, #3, blank glue tab]`; the **blank glue tab is always the fixed back cover** (the face glued into the journal). Imposition: outside face `[glue | content #1]`, inside face `[content #2 | content #3]`. **Overflow spills into multiple self-contained foldables** (3 content panels each; trailing slots padded blank) rather than scaling down — author's choice. Teacher instruction surfaced in the sidebar: print two-sided, flip on long edge, fold down the middle.
- **`Columns` is a worksheet-only setting and is hidden in foldable mode.** Worksheet columns = CSS `column-count` (browser flow/balance) on the flat page. The foldable packs blocks vertically into fixed fold regions, which CSS columns can't express, so the control is hidden there rather than shown-but-inert. The *right* long-term answer to "columns in a foldable" is structural authored columns (below), not a per-mode print toggle.

### Structural columns + variable block sizing

- **Columns are *content*, not a print mode.** A `ColumnsBlock` (2–6 `Column`s, columns-in-columns forbidden) lays blocks side by side in the document, so it renders identically on screen, in worksheet print, and inside a foldable panel — replacing the per-mode CSS `column-count` toggle. Per-column `width` is an optional unitless weight (absent ⇒ equal split). Renderer emits a CSS grid driven by a single `--columns-template` custom property so a media query can override track sizing (`@media screen ≤640px` collapses to one column; print/foldable stay multi-column). Full narrative in HISTORY.
- **Reflow-safe sizing only — no free canvas.** The whole variable-block-sizing arc is bounded to sizing that survives multi-output reflow (screen + worksheet print + journal foldable + future Chromebook/mobile): content-determined height, fractional widths, `min-height` floors that still grow, fixed image boxes with cropping. **Out of scope by deliberate decision:** arbitrary fixed pixel heights decoupled from content, and absolute/free-canvas positioning — both break the foldable's height measurement and print reflow (same reason the editor stays a vertical block stack, not a 2-D canvas).
- **One additive sizing fragment, custom-property-driven.** Optional `width` (fraction `(0,1]`) + `align` (`left|center|right`, default center = attribute-free) live in a shared `sizing.ts` fragment spread onto sizable blocks (image, math_block today; callout/problem have no editor mapping yet, so no fields there — YAGNI). Renderer adds a `block-sized` class + `--block-width`/`--block-height` custom properties + `data-block-align`; narrow screens relax widths to 100% (CSS), print keeps them. All additive/optional ⇒ **no `schemaVersion` bump, no `STORAGE_SCHEMA_VERSION` bump, no migration**; the runtime never reads any of it (purely presentational, additive data-attribute-contract entries in RUNTIME.md).
- **Image width: `Auto` (null) ≠ `100%` (1).** `null`/unset = natural size, never upscaled (the default). `width: 1` = a real "fill the container" width that upscales like any other percentage. They are distinct chips because conflating them (the original "Full"-means-null labeling) made a small image sit in an empty row looking broken.
- **Mismatched image dimensions CROP, never stretch.** When `width` × `height` disagree with the natural aspect ratio the rendered box uses `object-fit: cover` (center crop) — a photo-frame, not a funhouse mirror (author's choice over stretch). Height alone scales proportionally. `height` is in **rem** (scales with the print font-size config, like `Column.minHeight`).
- **Lead with reliable controls; drag gestures only where they prove out.** Column width drag-resize was built (a widget-decoration divider) and **cancelled** after it competed for the same gutter strip as the inner-block drag handle and didn't feel reliable under a real mouse — the **width presets remain the column-width system** (schema/serialize untouched, so a re-attempt stays possible; see git history for `columnResize.ts`). The lesson set the pattern for the rest of the arc: image resize ships popover chips/fields as the baseline *plus* drag-handles as an enhancement, and cell reserved-work-space (`Column.minHeight`) ships as a **toolbar dropdown only** (Auto / rem presets / numeric input via `setColumnMinHeight`), no drag.
- **A top-level columns container flows whole through the foldable — verified.** `measure.ts` extracts `.activity-section > *`, and a `columns` block is a single such child, so it becomes ONE `FlowItem` (its cells are never flattened into separately-paginated items) and `paginate` never splits it across a fold (an oversized container gets its own over-full panel, intact). Inner width resolves against the panel: the grid's `fr` tracks size against `.foldable-panel-content`'s fixed `panelWidthIn`, and the narrow-screen collapse (`@media screen ≤640px`) can't fire during measurement because it keys off the 1200px iframe viewport, not the element. Locked in by `foldable-measure.test.ts` (one item per top-level block; cells stay inside; `--columns-template` carries `fr` tracks) + a named `paginate` case, and proven in a real browser via the dev-only `/dev/foldable-columns` bench: a 2:1 container measured `gridTemplateColumns: 286.7px 143.3px` (ratio exactly 2), filling the 432px panel, in a single panel.

### Markdown import (Phase 1 polish)

- **One shared converter, deliberately standalone.** `lib/markdownToTiptap.ts` is `markdown → Tiptap JSONContent[]`, no editor/DOM dependency, exhaustively unit-tested. It's the same converter the queued **PDF import** reuses (a vision model emits the same markdown surface; the client converts it) — so it lives apart from the paste UI on purpose. Design context in `docs/design/pdf-import.md`.
- **App-only and additive — no schema/renderer/runtime/bundle/deploy.** The feature emits the exact JSONContent shapes `activityToTiptap` produces, so imported blocks round-trip through `tiptapToActivity` and render identically to authored ones. Nothing in the pure packages changes; no `bundle:renderer`, no `publish-activity` redeploy.
- **markdown-it, lazy-loaded, kept vanilla.** Chosen over a hand-rolled parser because the strategic consumer is *model-generated* markdown (PDF import, teachers drafting in an LLM): markdown-it IS CommonMark, so "the model emitted valid markdown my parser choked on" never happens. `await import('markdown-it')` keeps it off the initial editor bundle (it only runs behind the Import dialog). Config is stock (`html:false`, `linkify:false`); the custom constructs are resolved *around* it, **never** by patching its rule chain.
- **Two custom constructs resolved in the mapper; math resolved before markdown-it.** Blanks (`{{…}}`) and the `{checkpoint}` marker are read out of the post-parse text tokens — `{`, `}`, `|` aren't markdown-special (tables off), so they pass through verbatim and can't collide with markdown's grammar. **Math is the exception** and can't be done that way: LaTeX is full of backslashes (`\,`, `\frac`), underscores and carets that CommonMark's backslash-escape and emphasis rules corrupt *in the text tokens* (verified: `$\int_0^1 x\,dx$` came back as `…x,dx`). So `$…$` / `$$…$$` is lifted out of the **raw source first** (a `MATH_SCAN` pass that also matches and skips code spans, so `` `$x$` `` stays literal), swapped for a Private-Use-Area placeholder markdown-it forwards untouched, and re-expanded into `mathInline`/`mathBlock` in the mapper. Inline `$…$` uses a Pandoc-style guard (open `$` not before a space, close `$` not after a space/before a digit) so "$5 and $10" isn't math. A whole-paragraph `$$…$$` becomes a block `mathBlock`; inline elsewhere is `mathInline`. We only tokenize — no katex at import time, no rendering plugin.
- **Blank syntax reuses the editor sentinel `{{answer|alt1|alt2}}`** (same grammar as `Blank.ts`'s input rule and the runtime's `data-blank-answers` format). This **supersedes the illustrative `[[answer]]` sketch in `pdf-import.md`** — one mental model, one parser, no second blank syntax. An empty-answer `{{}}` stays literal text (a blank needs ≥1 char).
- **`{checkpoint}` on a heading → a checkpoint `sectionBreak`** (title = the heading text, marker stripped). Plain headings stay heading blocks (clamped to levels 1–3). A non-checkpoint section-break marker was YAGNI'd for v1.
- **Trigger is a header dialog, not paste-anywhere interception** — resolving the open question in ROADMAP.md ("inline vs. dialog"). Predictable, discoverable, never hijacks ordinary paste, and good for batch-pasting; smart-paste can layer on later. The button sits in the page header next to PublishControl (activity-level action, not an editor-toolbar control). Insert replaces a fresh/empty doc outright (no leading blank) and appends to a non-empty one.
- **A blank-bearing list flattens to one `fillInBlank` per item.** "Paste a numbered list of problems and get problem blocks" (ROADMAP.md:48) is the core use case; blanks can only live inside `fillInBlank` (per the PM schema), so the list marker is dropped and the editor re-numbers the resulting problem blocks. A list with no blanks stays an ordinary nested list. Inside a `fillInBlank` (content `(text|mathInline|blank)*`, no hardBreak) any break becomes a space.
- **Coverage is bounded by what the editor round-trips today; everything else degrades, never corrupts.** Covers headings, paragraphs, bold/italic/code, nested lists, blanks, checkpoint sections, `$…$`/`$$…$$` math, and `![alt](url)` images (an inline image is lifted out of its paragraph into a block-level image, splitting the paragraph around it in document order). Unsupported input (code fences, blockquotes, tables, raw HTML, links) flattens to text / drops the unsupported attribute with a deduplicated, human-readable warning surfaced in the dialog. Verified against the live ProseMirror schema (math KaTeX-renders, backslashes intact; image block carries src+alt).
- **The dialog's close is decoupled from FocusTrap — never `onDeactivate: onClose`.** First attempt tied close to the trap's `onDeactivate`; under React StrictMode the dev double-mount unmounts the trap, fires `onDeactivate`, and the dialog vanished the instant it opened. Close is now driven only by our own Escape-keydown + backdrop-mousedown handlers (`escapeDeactivates: false`, no `onDeactivate`) — same shape as `ImageEditPopover`. A StrictMode RTL regression test (`ImportMarkdownDialog.test.tsx`) renders under `<StrictMode>` and asserts the dialog stays open; reintroducing `onDeactivate: onClose` fails it. (`tabbableOptions: { displayCheck: 'none' }` lets the trap activate under jsdom — prod-safe, all elements visible.) RTL is the repo's first component-test infra.
- **The lifted editor instance is reported up but never nulled on cleanup.** The header Import button gates on the editor instance, lifted from `Editor` via `onEditorReady`. The effect reports only a non-null editor and never nulls on cleanup — nulling on the intermediate StrictMode/keyed-remount cleanup could leave the button stuck disabled. (A reported disabled-button was a stale dev server, not this; the hardening is defense-in-depth so the symptom can't recur.)
- **The accepted format is a single, drift-guarded source of truth.** `docs/markdown-import-format.md` is the authoritative spec and doubles as an LLM prompt; the canonical prompt is `MARKDOWN_IMPORT_AI_PROMPT`; the converter is the behavior. `markdownImportPrompt.test.ts` binds all three: the doc's prompt block must equal the constant byte-for-byte, every documented example must run through the live converter as promised, and the documented "unsupported" list must actually degrade. The Import dialog's **Copy AI prompt** button hands a teacher the prompt to paste into ChatGPT/Claude.
- **The AI prompt tells the model to fence its whole reply; the importer unwraps an outer fence.** LLMs render markdown in chat, so asking for "markdown" yields an uncopyable formatted preview. Wrapping the reply in a fenced code block makes the chat show a Copy button that yields raw markdown (the button strips the fence, so the importer never sees it). As a safety net the converter unwraps a whole-paste ```` ```markdown ```` / ```` ```md ```` fence before parsing — scoped to a markdown-tagged OUTER fence, so a plain ```` ``` ```` block mid-content is still flattened as code.

### Activity deletion — soft-delete via SECURITY DEFINER RPC (0008)

- **A client-side soft-delete (`update activities set deleted_at`) is fundamentally incompatible with the `deleted_at is null` SELECT policy, and no `WITH CHECK` tweak fixes it.** PostgreSQL enforces the table's SELECT policy against the **post-update row**, not just the UPDATE policy's own `USING`/`WITH CHECK`. `activities_select_own` requires `deleted_at is null`; the moment the soft-delete writes a timestamp, the new row is invisible to SELECT and the entire UPDATE is rejected with *"new row violates row-level security policy for table activities"*. Verified by isolating each clause against the live Postgres engine (synthetic temp tables, rolled back): the error persists even with the UPDATE policy at `WITH CHECK (true)`, and disappears only when the SELECT policy is dropped or its `deleted_at` clause removed. The original "the update policy permits an owner to set deleted_at" assumption (in `0002`'s comment and the first delete implementation) was never exercised until a real delete was attempted. `publish_activity` was unaffected purely because it never touches `deleted_at` — the one column the SELECT policy keys on.
- **Fix: a `SECURITY DEFINER` RPC `soft_delete_activity(p_activity_id)` (migration `0008`), not a relaxed SELECT policy.** The RPC runs as `postgres` (which has `BYPASSRLS`, and `BYPASSRLS` overrides `FORCE ROW LEVEL SECURITY`), authorizes via `can_edit_activity` (owner of a live activity — the same helper `publish_activity` uses, so Phase 3+ collaborator roles flow through for free), tombstones the row, and writes an `activity.delete` audit entry. The strict SELECT policy stays intact, so deleted rows remain hidden from clients exactly as before — chosen over dropping `deleted_at is null` from `activities_select_own` (the simpler migration) specifically to keep RLS, not just the app query, as the guard that hides soft-deleted rows. The client calls `supabase.rpc('soft_delete_activity', { p_activity_id })` instead of `.update()`.

### Order-independent blank groups (consume-once)

- **The problem the naive fix can't solve.** Factoring `(x + ☐)(x + ☐)` should accept `(2,3)` and `(3,2)` but reject `(2,2)`. Widening *both* blanks' acceptable sets to `{2,3}` accepts the swap but *also* wrongly accepts `(2,2)`/`(3,3)`. Rejecting the duplicate requires the blanks to coordinate so each correct answer is consumed once — which independent per-blank scoring can't express. So grouping is a real cross-blank concept, not an acceptable-answers tweak.
- **Be general where change is expensive, minimal where it's cheap.** Three layers with very different cost-to-change drove the design: the **runtime data-attribute contract** (every published page bakes it in; additive-only) is made *general now* — blanks carry a `data-blank-group="<id>"` and the runtime buckets by id, so richer grouping later needs no contract change; the **document/schema** layer (additive Zod fields are cheap here) stays *minimal* — one boolean `BlankToken.interchangeableWithPrevious`; the **editor + dashboard** (no external consumers) are *minimal*. Making the runtime contract general cost nothing extra (the runtime buckets by id either way), so generality landed exactly where retrofitting would be most painful.
- **Authoring is "grouped with previous", a boolean sugar that compiles to group ids.** A group = a maximal run of adjacent blanks each flagged `interchangeableWithPrevious` (the run's first blank anchors it; its own flag is ignored). The renderer ([`inline.ts` `computeBlankGroups`](../packages/renderer/src/inline.ts)) assigns the run's anchor blank id as the shared `data-blank-group`. Chosen over a block-level "any order" flag (grades wrong on a block mixing a factor pair with an unrelated blank) and over explicit cross-block group ids (needs a multi-blank-select UI that collides with the single-host popover's FocusTrap/selection model). The boolean is a *strict subset* of group ids, so upgrading to non-adjacent/cross-block grouping later is an additive `group?: string` field, never a breaking migration.
- **Consume-once = maximum bipartite matching, deterministic tiebreak.** [`runtime/blanks.ts` `scoreGroup`](../packages/renderer/src/runtime/blanks.ts) matches typed values to answer *slots* (each slot keeps its own answer list + strategy — we match against per-slot lists, never a flattened pool, which is what preserves consume-once) via Kuhn's augmenting path. Groups are 2–4 blanks, so cost is trivial. Values are processed in document order, so among equal duplicates the *last* one is the one left unmatched — a deterministic tiebreak (removes a latent flaky-test source). Empty members stay unscored (null), mirroring solo-blank behavior. `scoreBlanksInScope` buckets a scope's blanks by group id and is called by **both** `checkSection` and the submit `gatherResponses`, so check-time and submit-time verdicts can't diverge.
- **Grouped blanks resolve at check/submit, never on immediate-mode blur.** A grouped blank can't be judged in isolation (its group-mates may be empty/mid-typing), so scoring it on blur would flash a misleading ✗ on an answer that's correct once matched. `wireBlanks` skips the immediate-mode blur scorer for blanks with a group id; they resolve when a group-aware scope runs. An inherent, accepted constraint of the feature.
- **Client stays authoritative — no Edge Function change.** `ingest-submission` Zod-validates and stores the client's per-blank `correct` + `score` verbatim; the dashboard *displays* the stored verdict (it doesn't rescore). So grouping is contained to schema/renderer/runtime/editor, and the "make answer evaluation server-shareable" refactor stays parked for photo-grading (where a server/human actually regrades). Verified before building, since a server rescore would have silently failed `(3,2)≡(2,3)`.
- **Markdown marker is a leading tilde, `{{~3}}`.** Picked because `~` is captured by the existing `blankCanon` group (`[^{}|]+`) and rarely begins a real answer (a `|` prefix would collide with the alternate-answer separator). The importer strips it and sets the flag; on a problem's first blank it's a harmless no-op (the renderer ignores a leading flag). Kept in three-way sync (converter / `MARKDOWN_IMPORT_AI_PROMPT` / format doc) under the existing `markdownImportPrompt.test.ts` drift guard.
- **Dashboard shows the group's set, not the slot's own key.** A grouped blank's "answer key" column would otherwise show *that slot's* canonical answer, which can differ from the student's (correct) entry and read like a bug. `buildActivityIndex` attaches the group's full answer set to each member; the drill-down renders "x or y (any order)". The ✓/✗ was always right; this fixes the *explanation*.

### Reference panel (Phase 2)

Optional teacher-authored scaffold content (formula charts, vocab lists, conversion tables…). Shipped in three drops — A: persistence (serialize round-trip), B: rendering (screen + print), C: authoring. The schema field `ActivityDocument.referencePanel` pre-existed (Stage 9e forward-compat); no `schemaVersion` bump was needed anywhere.

- **Runtime is untouched — the panel renders OUTSIDE `.activity-section`.** The runtime's `init` walker scopes every query to `.activity-section`, so anything outside it is invisible to scoring / persistence / checkpoints. The panel renders as an `<aside>` / `<details>` outside the sections, so it needs zero runtime code and no `STORAGE_SCHEMA_VERSION` bump. `data-block-category="scaffold"` is therefore an analytics/CSS hook, NOT the scoping mechanism (category is a coarse discriminator, never the gate).
- **Screen = collapsible bottom-bar toolbar; print = static box at the top.** The author wanted on-screen reference to *follow the student* (a toolbar that scrolls with them) but print to be a plain worksheet box. One `referencePanel` drives both: a `position:fixed` bottom `<details>` (flex `column-reverse` pins the clickable bar at the viewport bottom and opens the panel *upward*) for screen, and a separate `.reference-print` box (hidden on screen, shown `@media print`) at the worksheet top. Two CSS presentations of the same rendered blocks; native disclosure ⇒ zero runtime JS.
- **Drag-to-reposition deferred; drag-to-RESIZE + scroll-clearance shipped via a sidecar.** Repositioning a movable modal-style panel stays deferred. But the on-screen panel gained two behaviors through a small self-contained sidecar script (`runtime/reference-panel.ts`, ~1 KiB, built as its own IIFE and inlined by `document.ts` ONLY when an activity has a panel — kept OUT of the scoring runtime so its strict init/render/state invariants stay untouched and panel-less pages ship none of it): (1) a `ResizeObserver` keeps the activity container padded by the panel's *live* height, so the `position:fixed` bottom panel can never permanently hide content — page and panel scroll independently and the last content always scrolls clear; (2) a top-edge drag handle resizes the open panel between a 96px floor and ~90vh, so a large reference never swallows the screen. Drag-to-resize was the author's explicit call: dragging a panel's edge to resize is a conventional, expected interaction, distinct from the ambiguous column-width drag that was cancelled (the "reliable controls" preference still holds for *that*). Size isn't persisted (resets on reload) and collapsing clears the dragged height; remembered open-state and click-outside-close remain unbuilt. This **revises the original "collapse-only, zero runtime JS" decision** — which earmarked exactly this ("addable later as a small runtime piece if students ask"). Two follow-up fixes after live testing: (a) the height cap + scroll live on the **body** (`max-height:60vh; overflow:auto`), NOT the panel — `flex-shrink`+`min-height:0` does not constrain a child against a `max-height`-only flex container, so panel-capping let the body keep full content height and overflow the page; the drag also drives the body's `max-height`, not the panel height. (b) The handle + body are wrapped in a `.reference-panel-content` flex column so the handle lands at the panel's **top** edge — the browser wraps `<details>` content (`::details-content`), making panel-level flex order of body-vs-handle unreliable across versions. Don't move the cap back to the panel or remove the wrapper.
- **Print inclusion is a single on/off toggle** (`PrintConfig.printReferencePanel`, default true). A teacher with a class set of printed charts turns it off; the on-screen toolbar is unaffected. Additive + defaulted → no schemaVersion bump. Granular per-component print selection deferred until asked for.
- **Authoring uses an ISOLATED `ReferencePanelEditor`, not a parameterized `<Editor>`.** The main editor is the repo's riskiest surface (single-host popover, `defining`/drag invariants, NodeViews); a separate component keeps that blast radius at zero. It registers a constrained extension set and runs the toolbar in `variant="reference"` (hides Problem / Blank / Section). The shared `Toolbar` gained only an additive `variant` prop — the main editor (default `'activity'`) renders exactly as before.
- **Panel fingerprint uses the editor's Tiptap JSON, not the serialized panel.** `tiptapToReferencePanel` mints fresh block UUIDs per call, so feeding the serialized `ReferencePanel` to change-detection would churn the autosave key (the standing "fingerprint Tiptap JSON, never serialized docs" rule). `ActivityEditor` holds `panelJson` + `panelTitle`, gates `changeKey` on *both* editors having reported (so the autosave baseline settles in a single transition — no spurious load-time save), and `panelFromEditor` reconstitutes the panel at save time (returns undefined for an empty panel, so an empty scaffold is never persisted).
- **`FillInBlank` / `Blank` are registered in the panel editor only to satisfy the schema.** The `column` cell's content expression references `fillInBlank` (which references `blank`), so a panel editor that offers columns can't build its ProseMirror schema without them. The reference toolbar still hides the Problem / Blank buttons, so a teacher can't *author* a blank in a panel; a pasted one renders inert (outside `.activity-section`). Cleaner alternatives (decoupling the shared `column` content spec, or dropping panel columns) were heavier, and the author had approved columns for formula-chart layouts.
- **StarterKit v3 bundles `Underline`** — discovered while debugging the panel editor's schema crash. The explicit `@tiptap/extension-underline` registration (added in the earlier underline commit) is redundant and warns "Duplicate extension names: underline". The panel editor relies on StarterKit's; the main editor's redundant copy is queued for cleanup.

### Inline vocabulary definitions (Phase 2)

Select text → "Define" → author a definition shown in a popover on the published page. Targets the math-vocabulary barrier (e.g. "factor" as noun vs. verb). Full design + rationale in [docs/design/vocabulary-definitions.md](design/vocabulary-definitions.md); the load-bearing decisions:

- **A mark, not a node.** The defined text is still text — it wraps, layers other marks, and is authored by selecting a run, not inserting a block. The definition *content* (rich, possibly with an image) is separate from the inline *trigger* (the mark).
- **Marks moved from a string enum to an object discriminated union** (`{ type, …attrs }`) so a mark can carry attributes — `definition` is the first attribute-carrying mark (`link`/`color` are latent future ones). Chosen over a minimal "string marks + one parallel field" approach: object marks are uniform (no heterogeneous `string | object` array), match Tiptap's own model, and future-proof. **No data migration** — the DB stores Tiptap JSON (already object marks); `ActivityDocument` is a transient publish artifact. A `z.preprocess` upgrades legacy bare-string marks (and, later, the legacy plain-`definition`-string mark) on read, so old documents parse with **no `STORAGE`/`schemaVersion` bump**.
- **Definition content is rich (text + inline math) + one optional image**, not a plain string — "a picture worth a thousand words." Content reuses the blank-hint stack: authored via the shared `InlineRichTextEditor`, stored as canonical `InlineNode[]` in the mark attr, pre-rendered by the renderer. The content's text run uses a **restricted inline type (simple marks only)** — this forbids nested definitions AND breaks the schema recursion that reusing `InlineNode` would create (`DefinitionMark → content → text → marks → DefinitionMark`). One image (not a full block document) was the deliberate scope: covers ~all vocab cases without a cramped block-editor-in-a-popover. Image reuses the existing `upload-image` / R2 path.
- **Runtime is a separate sidecar, not the scoring runtime.** Definitions appear anywhere inline text does (including the reference panel, outside `.activity-section`), are non-scored and non-persisted. So `runtime/definitions.ts` (~1.9 KiB, inlined only when a page has a definition) mirrors the reference-panel sidecar pattern: it manages its own popover and is kept out of the scoring runtime's strict init/render/state invariants. The renderer pre-renders content into a hidden `<template class="js-definition-content">` next to the span (KaTeX baked in); the sidecar **clones** it into the popover (never re-renders), falling back to a plain-text `data-definition` attribute. Additive to the frozen data-attribute contract.
- **Editing is a single root-level host** (`DefinitionPopoverHost`), mirroring `BlankPopoverHost` — never per-mark mounting (the standing reconciliation constraint). Mark-based (vs. the blank/image node hosts) via `getMarkRange`; the range start keys the popover so its draft reloads only on retarget, and commits land on close (a per-keystroke `extendMarkRange` would churn the undo stack).
- **Phase 4 glossary is account/tenant-scoped, not per-activity** (decided 2026-06-19). The eventual glossary lives in a tenant-level `glossary_entry` table (NOT `ActivityMeta`), resolved at publish; the mark references it by the already-reserved `glossaryKey`. The leaning is the **light "implicit-reuse"** model (the platform remembers a teacher's definitions and offers them back) over a heavyweight managed glossary; an official managed glossary, if built, is opt-in. **Auto-suggest** (find glossary terms in text) must be **suggest-don't-apply** — never silent/automatic at render — because of word-sense ambiguity, over-marking, teacher intent, and assessment integrity. None of this affects the shipped mark.
